# mygrep — распределённый `grep` с кворумом и concurrency

> **L4.2** — распределённая CLI-утилита поверх классического `grep`.
> Клиент режет вход на чанки, раздаёт их пулу серверов по HTTP, агрегирует
> результат и принимает итог при достижении кворума `N/2 + 1` успешных
> ответов. Внутри каждого сервера обработка строк параллельна — пул
> воркеров на горутинах с обменом через каналы.

## Возможности

| Флаг           | Аналог `grep` | Описание                                            |
|----------------|---------------|-----------------------------------------------------|
| `-e PATTERN`   | `-e`          | Регулярное выражение (по умолчанию RE2)             |
| `-F`           | `-F`          | Искать как фиксированную строку, без regex          |
| `-i`           | `-i`          | Игнорировать регистр                                |
| `-v`           | `-v`          | Инвертировать совпадение                            |
| `-n`           | `-n`          | Печатать **глобальные** номера строк                |
| `-c`           | `-c`          | Печатать только количество совпадений               |
| `-l`           | `-l`          | Печатать имя файла, если есть хотя бы одно совпадение |
| `--servers`    | —             | Список адресов узлов кластера, через запятую        |
| `--quorum N`   | —             | Кворум (по умолчанию `floor(N/2)+1`)                |
| `--timeout S`  | —             | Таймаут одного HTTP-запроса в секундах              |

Группы коротких флагов работают, как в GNU grep: `-Fni ERROR file.log`
эквивалентно `-F -n -i ERROR file.log`.

Код возврата клиента совместим с `grep`:

| Код | Значение                                            |
|-----|-----------------------------------------------------|
| `0` | Найдено хотя бы одно совпадение                     |
| `1` | Совпадений нет                                       |
| `2` | Ошибка (бад-regex, недостигнут кворум, нет файла…)  |

## Архитектура

```
            ┌───────────────────────────┐
            │            stdin / файл           │
            └──────────────┬────────────┘
                           ▼
                ┌─────────────────────┐
                │       mygrep        │ ← клиент
                │  split into N chunks│
                └──────────┬──────────┘
                           │  HTTP POST /process
        ┌──────────────────┼──────────────────┐
        ▼                  ▼                  ▼
 ┌───────────────┐ ┌───────────────┐ ┌───────────────┐
 │ mygrep-server │ │ mygrep-server │ │ mygrep-server │ ← узлы
 │  pool of N    │ │  pool of N    │ │  pool of N    │
 │  goroutines   │ │  goroutines   │ │  goroutines   │
 └──────┬────────┘ └──────┬────────┘ └──────┬────────┘
        └──────────────────┼──────────────────┘
                           ▼
              ┌────────────────────────┐
              │  агрегатор результатов  │
              │  • сортирует по chunk_id│
              │  • суммирует -c        │
              │  • ждёт кворум         │
              └───────────┬────────────┘
                          ▼
                       stdout
```

Каналы и горутины используются **на двух уровнях**:

1. **Внутри сервера**: пул воркеров (`runtime.NumCPU()` по умолчанию)
   читает строки из `jobs` chan и пишет совпадения в `out` chan; затем
   результаты сортируются по индексу строки, чтобы итог не зависел от
   порядка завершения воркеров.
2. **Между серверами**: клиент стартует горутину на каждый сервер и
   собирает ответы в общий `chan` с дедлайном по `--timeout`.

## Быстрый старт

```bash
# 1. собрать бинарники в ./bin
make build

# 2. поднять 3 сервера, выполнить поиск, остановить — одной командой
make run                                  # ищет ERROR в examples/data/access.log
PATTERN="status=500" make run             # любой свой паттерн

# 3. убедиться, что вывод бит-в-бит совпадает с системным grep
make compare
```

Минимальные требования: **Go 1.22+**, `bash`, `curl`, `awk`.

## Использование клиента `mygrep`

```
mygrep --servers host1:port,host2:port,... -e PATTERN [flags] [file]
```

Если `file` не указан — вход читается из `stdin`.

### Примеры командной строки

```bash
# Прямой поиск ошибок в логе через 3 ноды
./bin/mygrep --servers 127.0.0.1:9101,127.0.0.1:9102,127.0.0.1:9103 \
             -F -n -e ERROR examples/data/access.log

# То же из stdin
cat examples/data/access.log | ./bin/mygrep \
    --servers 127.0.0.1:9101,127.0.0.1:9102,127.0.0.1:9103 \
    -F -e ERROR

# Считаем 5xx по регулярке
./bin/mygrep --servers 127.0.0.1:9101,127.0.0.1:9102,127.0.0.1:9103 \
             -c -e 'status=5\d\d' examples/data/access.log

# Игнор регистра + инверсия
./bin/mygrep --servers 127.0.0.1:9101,127.0.0.1:9102,127.0.0.1:9103 \
             -Fvi -e info examples/data/access.log

# Явный кворум: достаточно одного ответа
./bin/mygrep --servers a:1,b:2,c:3 --quorum 1 -e ERROR file.log
```

## Сервер `mygrep-server`

```
mygrep-server [-addr :PORT] [-workers N]
```

| Флаг        | Значение по умолчанию | Описание                                         |
|-------------|-----------------------|--------------------------------------------------|
| `-addr`     | `:8080`               | Адрес прослушивания (`host:port` или `:port`)    |
| `-workers`  | `runtime.NumCPU()`    | Кол-во горутин-воркеров на один запрос           |

Эндпоинты:

- `POST /process` — основной обработчик. Тело — `ProcessRequest` в JSON.
- `GET  /healthz` — лайвнесс-проба, используется `scripts/run_local.sh`.

Корректно завершается по `SIGINT` / `SIGTERM` (graceful shutdown 5 с).

## Как работает кворум

1. Вход режется на `N = len(--servers)` непрерывных чанков по строкам.
   Каждому чанку клиент присваивает `chunk_id` и `start_line` (1-based),
   чтобы итоговый порядок строк не зависел от порядка ответов сети.
2. Клиент шлёт `POST /process` параллельно всем `N` серверам.
3. Ответы собираются с дедлайном `--timeout`. Успешным считается
   HTTP-200 с пустым полем `error`.
4. **Кворум достигнут**, если число успешных ответов
   `≥ floor(N/2)+1` (или значения `--quorum`, если указано явно).
5. Итог собирается из успешных ответов в порядке `chunk_id`.
   Если кворум **не достигнут**, в `stderr` печатается предупреждение
   с перечнем ошибок по серверам, а код возврата клиента — `2`.

> **Важно.** Чанки не реплицируются: каждый отдаётся ровно одному
> серверу. Поэтому при сбое одного из узлов соответствующий чанк
> попадает в "потерянные", даже если по числу успешных ответов кворум
> формально достигнут. Это соответствует постановке L4.2 и явно
> подсвечивается клиентом в выводе stderr.

## Примеры

В каталоге `examples/data/`:

- `access.log` — синтетический access-лог веб-сервиса с уровнями
  `INFO/WARN/ERROR/DEBUG`.
- `words.txt` — список слов в смешанном регистре.

Готовые сценарии — в каталоге `scripts/`:

| Скрипт                         | Что делает                                                                       |
|--------------------------------|----------------------------------------------------------------------------------|
| `scripts/run_local.sh`         | Поднимает 3 сервера, запускает `mygrep`, корректно останавливает сервера.        |
| `scripts/compare_with_grep.sh` | Прогоняет 11 кейсов на разных флагах и сверяет вывод с системным `grep` (PASS/FAIL). |
| `scripts/bench.sh`             | Генерирует 200k строк и сравнивает время выполнения mygrep vs `grep`.            |

Пример:


## Сравнение с системным `grep`

`scripts/compare_with_grep.sh` гонит набор кейсов и для каждого сверяет
вывод побайтно:

```bash
$ ./scripts/compare_with_grep.sh
== Сравнение mygrep vs системного grep ==
   data: .../examples/data/access.log
   servers: 127.0.0.1:9201,127.0.0.1:9202,127.0.0.1:9203

  [PASS] fixed ERROR (default)                         exit=(grep=0 mygrep=0)
  [PASS] fixed ERROR -n                                exit=(grep=0 mygrep=0)
  [PASS] fixed ERROR -c                                exit=(grep=0 mygrep=0)
  [PASS] fixed status=200 -v                           exit=(grep=0 mygrep=0)
  [PASS] regex ^2025-05-25T10:00:0[1-3]                exit=(grep=0 mygrep=0)
  [PASS] regex status=(500|502) -n                     exit=(grep=0 mygrep=0)
  [PASS] ignore case 'error' -i                        exit=(grep=0 mygrep=0)
  [PASS] no matches (DOESNOTEXIST)                     exit=(grep=1 mygrep=1)
  [PASS] words fixed 'lima' -i                         exit=(grep=0 mygrep=0)
  [PASS] words regex ^[A-Z]                            exit=(grep=0 mygrep=0)
  [PASS] words count letters -c                        exit=(grep=0 mygrep=0)

== Итог: PASS=11  FAIL=0 ==
```

Запуск с `-v` показывает diff при расхождениях:
`./scripts/compare_with_grep.sh -v`.

## Запуск в Docker

В репозитории есть `Dockerfile` (multi-stage, target `server` и `client`)
и `docker-compose.yml`. Пример:

```bash
docker compose up --build -d server1 server2 server3

docker compose run --rm client -F -n -e ERROR /data/access.log
```

`examples/data/` монтируется внутрь клиента как `/data`.

## Тесты и линтеры

```bash
make test    # go test -race ./... — юнит и интеграционные
make vet     # go vet ./...
make fmt     # gofmt -w
```

Покрытие:

- `internal/chunk`  — разбиение входа, нормализация переносов, краевые
  случаи (пустой вход, строк меньше чем серверов).
- `internal/grep`   — fixed/regex/ignore-case/invert, плохие регулярки.
- `internal/server` — HTTP-обработчики через `httptest`, проверка
  глобальных номеров строк, режима `-c`, `-v`, плохой regex.
- `internal/client` — кворум достигнут / не достигнут, упорядоченность
  результатов, `-c`, обработка падающих серверов.
- `cmd/mygrep`      — разворачивание групп коротких флагов.
